### Decision: Labeling approach — weak supervision

**What:** we don't have ground-truth bug labels and won't manually label.
Instead we treat bug-fix commits as a proxy and accept noisy labels.

**Why:** there is no `is_bug` column anywhere in git or GitHub. Most real-world
ML works this way (called *weak supervision*). The classic algorithm for this
in software-engineering research is **SZZ** (Śliwerski–Zimmermann–Zeller).

Logical chain:
```
bug-fix commit (proxy)
   ↓
files modified in that commit (events.file_path)
   ↓
those files had a bug — at least the one being fixed
```

So our approximate label for "this file had a bug" = "a bug-fix commit later
modified it." The fix stands in for the bug, since the fix is what's visible
in git.

**Alternatives rejected:** manual labeling (not feasible at scale);
giving up on supervised learning (defeats the project goal).

### Decision: Primary signal for bug-fix detection — PR labels

**What:** identify a commit as a bug-fix if it links to a PR whose labels
include something matching `bug` (case-insensitive, possibly prefixed like
`type/bug`, `kind/bug`).

**Why:** of the four signals considered, PR labels have the highest precision
(human-applied, deliberate). Training on noisy labels degrades the model more
than training on a smaller clean set.

| Signal                                      | Precision | Recall | Notes                                             |
|---------------------------------------------|-----------|--------|---------------------------------------------------|
| PR labeled `bug` (or similar)               | High      | Low    | Human-applied; depends on PR-link being parseable |
| Issue closed by commit, where issue is bug  | Medium    | Medium | Many issues are feature requests, not bugs        |
| Commit message keywords (`fix`, `bug`)      | Low       | High   | Catches `"fix typo"`, `"fix tests"` — noisy       |
| `is_revert` flag                            | Special   | —      | Marks reverted commits, not the bug-fix itself    |

**Alternatives rejected:** commit-message keywords as primary (too noisy:
`"fix typo"`, `"fix tests"` aren't bug fixes). Issue-link as primary (too
noisy: many issues are feature requests).

### Decision: Train framing — temporal split with 90-day label window

**What:** train with a temporal split, not random.

```
For each file in the repo at cutoff T:
  features  = stats from events with committed_at <  T
  label     = 1 if any bug-fix commit in [T, T + 90d] touched the file
```

**Why:** a random split lets the model see future data while learning, which
is data leakage. Example: if `auth.py` is fixed on Jan 15 and `num_recent_fixes`
is computed over the full history, that count includes the Jan 15 fix — the
model peeks at the answer. At inference time you only have history up to *now*,
so training must mimic that.

**Window length — 90 days:** shorter windows miss slow-fixed bugs (label
false negatives); longer windows need older training data and slow iteration.
60 was considered; 90 matches common practice in literature.

**Alternatives rejected:** random train/test split (causes leakage); 60-day
window (too short for slower fixes); no window cutoff at all (label undefined).
